Техническое задание
Проект: «Банковская система с клиент-серверным взаимодействием»
1. Цель проекта
  Создать консольное приложение на языке C++, моделирующее работу упрощённой банковской системы с возможностью создания и управления аккаунтами пользователей через сетевое взаимодействие по принципу Клиент–Сервер.
2. Общая архитектура
  Система должна состоять из двух основных компонентов:
  Серверное приложение — отвечает за хранение данных пользователей, обработку запросов клиентов и выполнение банковских операций.
  Клиентское приложение — подключается к серверу по сети, отправляет команды и получает результаты операций.
К одному серверу одновременно могут подключаться несколько клиентов (не менее трёх).
3. Основной функционал
  3.1. Функции сервера
    Принимать подключения от клиентов.
    Обрабатывать запросы, поступающие от клиентов.
    Хранить данные о пользователях и их счетах.
    Выполнять операции:
    создание нового аккаунта (регистрация);
    авторизация пользователя;
    просмотр баланса;
    пополнение счёта;
    снятие средств;
    перевод денег между пользователями.
    Возвращать клиенту результат выполнения каждой операции.
    Вести журнал (лог) всех подключений и действий.
  3.2. Функции клиента
    Устанавливать соединение с сервером.
    Предоставлять пользователю интерфейс (через консоль) для ввода команд.
    Отправлять команды серверу в установленном формате.
    Получать и отображать ответы сервера.
    Поддерживать следующие пользовательские действия:
    регистрация нового аккаунта;
    вход в существующий аккаунт;
    просмотр текущего баланса;
    пополнение счёта;
    снятие средств;
    перевод средств другому пользователю.
4. Требования к реализации
  Язык программирования: C++.
  Сетевое взаимодействие: сокеты (TCP/IP).
  Сервер должен быть многопоточным ИЛИ уметь обрабатывать несколько клиентов последовательно.
  Данные о пользователях должны храниться в оперативной памяти (для продвинутого варианта — сохраняться в файл).
  Приложение должно корректно обрабатывать ошибки (неверный ввод, отсутствие соединения, недостаточно средств и т. д.).
  Код должен быть структурирован и разделён по файлам (заголовочные и исходные файлы).
5. Сценарий демонстрации
  Запустить серверное приложение.
  Запустить три экземпляра клиентского приложения.
  Каждый клиент выполняет следующие действия:
  создаёт собственный аккаунт;
  выполняет несколько операций (пополнение, снятие, перевод);
  проверяет баланс.
  На стороне сервера в консоли отображается информация о всех подключениях и произведённых операциях.
6. Критерии оценки
  Критерий	Описание
Работоспособность	--> Приложение корректно выполняет все заявленные операции
Архитектура	--> Код разделён на клиент и сервер, использованы сокеты
Логика работы	--> Реализована регистрация, авторизация и базовые банковские операции
Обработка ошибок	--> Корректная реакция на неправильный ввод и сетевые ошибки
Демонстрация	Возможность одновременной работы нескольких клиентов
Дополнительно (по желанию)	
Сохранение данных в файл, использование потоков, простая авторизация
